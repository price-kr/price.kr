# 📄 Product Requirements Document (PRD): 집단 지성 오픈소스 단축 URL 서비스 (가칭: K-Shortcut)

## 1. 프로젝트 개요 (Overview)
본 프로젝트는 사용자가 브라우저 주소창에 `단어.가격.kr`(예: `만두.가격.kr`)을 입력하면, **DNS + Webapp 2단계 구조**를 통해 커뮤니티가 투표하여 선정한 최적의 목적지(최저가 쇼핑몰, 트렌드 페이지 등)로 리다이렉트해 주는 서비스입니다.

**동작 방식:**
1. `만두.가격.kr` 접속 → 와일드카드 DNS가 `가격.kr` 서버로 라우팅
2. 서버가 서브도메인(`만두`)을 추출하여 `가격.kr/만두`로 리다이렉트
3. `가격.kr/만두` 페이지에서 매핑된 최종 목적지 URL로 리다이렉트

중앙 집중형 관리 방식을 탈피하여, GitHub의 Issue와 PR(Pull Request), 그리고 커뮤니티 투표(Emoji)를 통해 100만 개의 한국어 키워드 매핑 데이터를 자동으로 구축하고 유지보수하는 완전 오픈소스 지향 파이프라인을 구축하는 것을 목표로 합니다.

## 2. 핵심 목표 (Objectives)
* **초고속 리다이렉트:** 엣지(Edge) 네트워크를 활용하여 사용자가 지연(Latency) 없이 목적지에 도달하도록 합니다.
* **100만 개 키워드 확장성:** Git 저장소의 병합 충돌(Merge Conflict)을 방지하고 성능 저하가 없는 최적의 데이터 디렉토리 구조를 설계합니다.
* **유지보수 제로(0) 자동화:** 관리자의 개입 없이, 사용자의 제안과 커뮤니티의 투표만으로 시스템이 스스로 업데이트되는 생태계를 구축합니다.

## 3. 타겟 사용자 (Target Audience)
* **엔드 유저 (End User):** 특정 상품의 최저가를 찾거나 특정 키워드에 대한 가장 유용한 링크에 빠르게 접속하고 싶은 일반 사용자.
* **기여자 (Contributor):** 유용한 링크를 공유하고 싶거나, 오픈소스 생태계에 Issue 작성 및 투표로 가볍게 참여하고 싶은 커뮤니티 멤버.

## 4. 기술 스택 (Tech Stack)
* **프레임워크 & 호스팅:** Next.js, Vercel (Edge Network)
* **DNS:** 와일드카드 DNS (`*.가격.kr` → `가격.kr` 서버로 라우팅)
* **라우팅 처리:** Next.js Proxy/Middleware (서브도메인 추출 → `가격.kr/{keyword}`로 리다이렉트)
* **캐시 및 스토리지:** Vercel KV (Redis) 기반의 초고속 L2 캐싱
* **데이터 버전 관리:** GitHub Repository (JSON 파일 기반)
* **CI/CD 및 자동화:** GitHub Actions (Issue 파싱, PR 자동 생성, 투표 집계, 병합, Vercel KV 동기화)

## 5. 핵심 기능 (Core Features)

### 5.1. DNS Resolver + Webapp 2단계 라우팅

#### 1단계: DNS → 서브도메인 리다이렉트
* `*.가격.kr` 와일드카드 DNS를 설정하여 모든 서브도메인 요청을 `가격.kr` 서버로 라우팅합니다.
* Next.js Proxy(`proxy.ts`)에서 요청의 `Host` 헤더를 파싱하여 서브도메인을 추출합니다.
* 한글 서브도메인의 경우 브라우저가 전송하는 **퓨니코드(Punycode, 예: `xn--o39a244b`)**를 디코딩하여 원래 한글 키워드로 변환합니다.
* 추출된 키워드를 경로로 하여 `가격.kr/{keyword}`로 `302 Redirect`합니다.
  * 예: `만두.가격.kr` → `가격.kr/만두`

#### 2단계: Webapp → 최종 목적지 리다이렉트
* `가격.kr/{keyword}` 페이지(Next.js 동적 라우트 `app/[keyword]/page.tsx`)에서 해당 키워드의 매핑 정보를 조회합니다.
* Vercel KV에서 키워드를 키(Key)로 검색하여 타겟 URL을 찾고, `301 Redirect`를 수행합니다.
* 매핑이 없는 키워드의 경우, 검색 페이지 또는 '새 단어 제안하기' 안내 페이지를 표시합니다.

### 5.2. 확장 가능한 100만 개 데이터 구조
Git의 파일 개수 및 용량 제한, PR 충돌을 피하기 위해 **초성 폴더 > 첫 글자 파일** 단위로 JSON을 분산 저장합니다.
```text
📦 data
 ┣ 📂 ㄱ (초성)
 ┃ ┣ 📜 가.json  # { "가방": "url", "가구": "url" }
 ┃ ┗ 📜 고.json
 ┣ 📂 ㅁ (초성)
 ┃ ┣ 📜 마.json
 ┃ ┗ 📜 만.json  # { "만두": "url" }
```

### 5.3. 커뮤니티 기반 자동화 파이프라인 (The Auto-Merge Flow)
1. **제안 (Issue):** 사용자가 정해진 양식의 GitHub Issue(키워드, 타겟 URL)를 생성합니다.
2. **PR 자동 생성:** GitHub Actions 봇이 Issue를 읽고 유효성을 검증한 뒤, 해당 키워드의 JSON 파일을 수정하는 PR을 자동으로 엽니다.
3. **투표 (Voting):** 커뮤니티가 해당 PR에 👍(찬성) 또는 👎(반대) 이모지를 남깁니다.
4. **자동 병합 (Auto-Merge):** 정해진 조건(예: 찬성 10개 이상, 반대보다 많음)을 충족하면 봇이 `main` 브랜치에 PR을 병합합니다.
5. **Vercel KV 동기화:** `main` 브랜치 업데이트를 감지하여, 변경된 키워드를 퓨니코드로 변환 후 Vercel KV에 반영합니다.

## 6. 사용자 플로우 (User Flow)

* **링크 사용자:** 주소창에 `만두.가격.kr` 입력 ➔ 와일드카드 DNS로 `가격.kr` 서버 도달 ➔ Proxy가 서브도메인 추출 후 `가격.kr/만두`로 리다이렉트 ➔ Webapp이 Vercel KV에서 URL 매핑 확인 ➔ 네이버 쇼핑 최저가 페이지로 이동
* **데이터 기여자:** GitHub 방문 ➔ '새 단어 제안하기' Issue 작성 ➔ 봇이 생성한 PR에 커뮤니티 투표 진행 ➔ 통과 시 실제 서비스에 반영

## 7. 제약 사항 및 리스크 (Constraints & Risks)
* **어뷰징 방지:** 스팸성 링크나 악성코드 사이트 등록을 막기 위해 초기에는 특정 도메인(예: 네이버 쇼핑, 쿠팡, 구글 트렌드 등)만 허용하는 화이트리스트 정책이 필요할 수 있습니다.
* **투표 조작:** 다중 계정을 통한 어뷰징 투표를 방지하기 위한 GitHub Actions 수준의 필터링 로직(가입일, 기여도 기반 등) 고도화가 필요합니다.
